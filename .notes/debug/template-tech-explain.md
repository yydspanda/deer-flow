# 技术原理讲解模板

> 目标：讲完之后，读者能预测这个技术在新场景下会怎样。
> 读者到达水平：能给别人讲（Level 2）+ 能改它（Level 3）。

---

## 必须有的（缺一个就不完整）

### ① 痛点引入 — "没有它会怎样"

不要说"它是什么"，要展示"没有它会怎样"。

用一个具体的失败场景开场，让读者感受到痛。痛点必须在解释方案之前出现，不能倒过来。

```
反面：TodoMiddleware 是一个管理待办列表的中间件，它继承自...

正面：你让 Agent 做一个 5 步任务。它做到第 3 步，对话太长，
      系统压缩了历史。Agent 忘了第 4 步是什么。
      它开始重复做第 1 步。这就是为什么需要上下文丢失检测。
```

### ② 心智模型 — 一句话类比

用日常类比，不用技术术语。测试标准：读者能不能用它猜对一个从未见过的行为。

心智模型的标准：技术上不完全对，但预测结果是对的。只有一句话，不超过 15 个字。

```
Git branch = 便利贴贴在 commit 上 → 预测：删便利贴不删 commit
write_todos = 白板，每次擦掉重写   → 预测：并行调用会冲突
熔断器 = 电闸                      → 预测：跳闸后完全断电，不慢慢降级
DeerFlow 中间件 = 传送带            → 预测：before_* 正序，after_* 反序
```

### ③ 分层归属图 — "这行代码是谁写的"

读者最大的困惑：我读的这段代码是框架提供的？还是项目自己写的？

画一个 box 分层图，每层标注文件路径。明确告诉读者：哪些不需要关心，哪些需要理解和可能修改。

```
┌─────────────────────────────────────┐
│ 框架/SDK 提供（你不需要写）           │
│   ① xxx 工具定义                      │
│   ② xxx 注入                          │
│   ③ xxx 检测                          │
├─────────────────────────────────────┤
│ 项目自己增加（文件路径）               │
│   ④ xxx 功能                          │
│   ⑤ xxx 功能                          │
└─────────────────────────────────────┘
```

### ④ 数据流图 — 一条链路从头追到尾

选一个具体场景，每一步标注：做了什么（动词）+ 在哪（文件:行号）+ 为什么在这（设计原因）。

必须是 ASCII 图（可以 grep，可以放注释里）。只追踪一条链路，不要同时画多条。

```
用户输入
  │
  ├─ [文件:行号] 做了什么
  │                → 为什么在这：设计原因
  │
  ├─ [文件:行号] 做了什么
  │                → 为什么在这：设计原因
  │
  └─ 输出结果
```

### ⑤ 核心机制 — "靠什么做到的"

数据流讲"经过哪些步骤"（WHEN/WHERE），案例追踪讲"发生了什么"（WHAT），
但没有人讲"靠什么做到的"（HOW）。这一层补上。

列出系统依赖的关键抽象/原语/数据结构，每个用 3 句话讲清楚：
它是什么 → 怎么工作 → 为什么用它。

常见机制类型：
- 状态更新原语（Command、setState）
- 路由/控制流（jump_to、goto）
- 数据结构（state schema、消息类型）
- 合并/冲突规则（reducer、去重）
- 注册/接线（组件怎么被插入系统）
- 钩子/回调（生命周期钩子）

```
例：

#### Command(update={...})

LangGraph 的状态更新原语——工具直接修改 state 字段，不走函数返回值。

怎么工作：write_todos 返回 Command(update={"todos": [...], "messages": [...]})
LangGraph 拦截这个返回值，把 update 里的字段直接写入 state。

为什么用它：
  → 如果用 return 值，todos 会变成 ToolMessage 的文本（字符串），不是结构化数据
  → Command 让 todos 保持 list[dict] 格式，后续中间件能直接读

例子：write_todos([{status:"pending", content:"搜索"}]) → state["todos"] 直接变成这个列表
```

### ⑥ 案例追踪 — 用具体输入输出展示过程脉络

用一个具体场景，展示每一步的输入→处理→状态变化→输出。
每一步标注"靠的机制"，关联到⑤核心机制。

不是贴测试代码，是写一段时序举例，让读者跟着看状态怎么变的。可以嵌在注释里、docstring 里、文档里。

```
请求 1: 502 错误 → 重试 3 次都失败 → failure_count=1 → 返回"暂时不可用"
         ↑ 靠的机制：指数退避

请求 2: 502 错误 → 重试 3 次都失败 → failure_count=2 → 返回"暂时不可用"
请求 3: 502 错误 → 重试 3 次都失败 → failure_count=3 → 熔断器跳闸(OPEN)
         ↑ 靠的机制：熔断器三态转换

请求 4: 熔断中 → 直接返回"熔断器跳闸"（不调 API）
... 10 秒后 ...
请求 5: 超时到期 → 转入 HALF_OPEN → 放一个探测请求
         ↑ 靠的机制：_check_circuit() 超时检测
  ├─ 成功 → 熔断器恢复(CLOSED)，failure_count 重置为 0
  └─ 失败 → 熔断器重新跳闸(OPEN)，重新计时 10s
```

## 锦上添花的（有更好，没有也不影响主线）

### ⑦ 常见坑 — "出问题时会怎样"

Happy path 谁都能理解。真正区分"读过"和"理解"的是边界条件。每个坑对应一个设计决策。

```
坑：摘要截断 write_todos    → 设计：before_model 上下文丢失检测
坑：Agent 提前说"做完了"     → 设计：after_model + jump_to="model"
坑：Agent 真的做不完死循环   → 设计：_MAX_COMPLETION_REMINDERS=2
坑：模型并行调 2 次白板      → 设计：基类 after_model 并行检测
```

### ⑧ 设计决策 — "为什么这样而不是那样"

只讲做了什么是不够的，要讲为什么没选另一个方案。
注意：不要重复核心机制已经讲过的内容，只补充核心机制没回答的"为什么"。

```
例：为什么 write_todos 每次调用替换整个列表，而不是增量更新？
→ 因为 LLM 不擅长部分更新（可能漏掉 status 字段），
  整体替换更简单可靠。代价是不能并行调用。
```

### ⑨ 对比表 — 和类似方案的区别

```
| 特性              | 方案 A        | 方案 B            |
|-------------------|--------------|-------------------|
| 上下文丢失检测     | ✅           | ❌                |
| 过早退出预防       | ✅           | ❌                |
| 并行调用保护       | ✅           | ❌                |
```

## 渐进深度 — 同一个概念，三个粒度

每个技术点都要有三个版本，读者按需选择：

```
10 秒版（一句话）：
  给第一次听说的人建立初步印象

1 分钟版（一段话 + 一张图）：
  给需要快速了解的人建立心智模型

10 分钟版（完整链路 + 案例 + 边界）：
  给需要真正理解的人建立预测能力
```
