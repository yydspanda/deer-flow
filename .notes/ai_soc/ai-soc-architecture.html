Memory × Dedup
LLM 看记忆，自己判断重复
DeerFlow 已有完整的记忆系统，Agent 研判完 → 自动提取 fact → 下次告警进来 → fact 注入 system prompt → LLM 自己认出重复
✦
每条规则一个独立记忆 · DeerFlow 原生支持
DeerFlow 按 (user_id, agent_name) 隔离记忆，存储为独立文件：
user_dir/{user_id}/memory/{agent_name}.json
在安全场景下，agent_name = 规则ID，每条规则有自己的 memory.json、自己的 facts 列表。
好处：EDR-D盘规则的记忆不会污染暴力破解规则的记忆，各自积累各自的研判经验。
记忆文件结构
memory/
  rule-edr-suspicious-process.json  ← EDR 可疑进程规则的记忆
  rule-brute-force.json              ← 暴力破解规则的记忆
  rule-c2-communication.json         ← C2 通信规则的记忆
  rule-lateral-movement.json        ← 横向移动规则的记忆
  ...
  每个文件里是独立的 facts 列表，互不干扰
MemoryMiddleware 在 Agent 执行完后自动工作：
① after_agent 过滤对话 → ② 入队(30s去抖) → ③ LLM 提取 facts → ④ 存到该规则的 memory.json
下次同规则告警进来 → format_memory_for_injection() 加载该规则的 facts 注入 system prompt
agent_name = 规则ID → 天然隔离 storage.py — 按 agent_name 存独立文件 updater.py — LLM 提取 facts prompt.py — 注入该规则的记忆
01
去重不需要新代码，LLM 看记忆就能判断
关键洞察：DeerFlow 的记忆是给 LLM 看的，不是给机器查的索引。
Agent 的 system prompt 里已经注入了所有高置信度 facts。
新告警进来 → Agent 看到 fact 里写过 "svchost_update.exe 是后门 INC-042" → 自己就能判断重复。
例子：EDR 可疑进程规则（rule-edr-suspicious-process）
⬇ 第一次告警
EDR 触发规则 rule-edr-suspicious-process：{process: "svchost_update.exe", path: "D:\\tools\\", host: "server-03"}
→ 加载该规则的 memory.json → 空，没有相关 fact → Agent 研判 → "后门，伪装系统更新" → 开了 INC-042
→ MemoryMiddleware 自动提取 fact → 存进 rule-edr-suspicious-process.json：
"content": "server-03 上发现 D:\\tools\\svchost_update.exe 后门，伪装系统更新 → INC-042"
"category": "ioc",  "confidence": 0.95
⬇ 第二次告警（同一规则，EDR 又报了）
EDR 再次触发规则 rule-edr-suspicious-process：{process: "svchost_update.exe", path: "D:\\tools\\", host: "server-04"}
→ 加载 rule-edr-suspicious-process.json → 注入 system prompt：
Facts:
- [ioc | 0.95] server-03 上发现 D:\\tools\\svchost_update.exe 后门，伪装系统更新 → INC-042
→ Agent 看到 system prompt 里这条 fact → "svchost_update.exe？我见过，INC-042 那个" → 合并到 INC-042，不新建
↑ 记忆范围限定在该规则内 — 暴力破解规则的告警不会看到 EDR 规则的 fact，各自独立
→ 零代码改动，只定制 Prompt 让 LLM 提取安全相关的 facts（进程名/IP/UM/IOC/Incident编号）
唯一需要做的：定制 MEMORY_UPDATE_PROMPT
在安全场景的 fact 提取指令里加上：
  • fact.content 必须包含 关键实体（进程名/文件路径/IP/UM账号/域名）
  • fact.content 必须包含 Incident 编号（如 → INC-042），这样下次 Agent 看到 fact 就知道合并到哪个 Incident
  • category 使用安全分类：ioc / asset / attack_pattern / affected_range / cve
vs
为什么不造轮子：记忆 vs 向量数据库 vs 手写索引
对比项 DeerFlow 记忆（本次方案） 手写 artifacts 索引 Qdrant 向量检索
额外代码 ✅ 零，只改 Prompt 要写索引+查询+维护 要写 embedding 管道
匹配能力 ✅ LLM 理解语义，能关联变形 精确匹配，变形就漏 相似度匹配，但不可控
可解释性 ✅ fact 是自然语言，人和 LLM 都能看 结构化，可审计 ❌ embedding 黑盒
基础设施 ✅ 零，memory.json 已有 内存索引即可 ❌ 要部署 Qdrant + 模型
积累方式 ✅ 研判完自动提取，越用越准 也要写提取逻辑 要写 embedding 管道
