# AI 时代工程师学习方法论

> 核心观点：AI 能写代码，但不能做架构决策、不能判断边界条件、不能理解业务上下文。
> 你的价值不在"会写代码"，而在"能判断代码对不对、架构合不合理"。

---

## 一、学架构决策：三问法

每读一个模块，只问三个问题：

### 1. 它解决什么问题？

没有它会怎样？如果没有这个模块，代码会变成什么样？

### 2. 它放弃了什么？

为什么不选另一个方案？每个设计决策都是取舍，知道它放弃了什么才算真正理解。

### 3. 它在什么条件下会崩？

边界和极限在哪？隐式依赖是什么？排序错了会怎样？

### DeerFlow 实战案例

#### 案例 1：中间件链

| 问题 | 答案 |
|------|------|
| 解决什么？ | Agent 的横切关注点（安全、记忆、审计）如果不抽成中间件，就散落在业务代码里，加一个功能要改 10 个地方 |
| 放弃了什么？ | 放弃了洋葱模型的"请求和响应走同一条路"的对称美感。因为大部分中间件只关心请求（before）或只关心响应（after），强制对称是过度设计 |
| 什么条件会崩？ | 中间件之间有隐式依赖（ThreadData 必须在 Sandbox 前），排序错了不会报错但会运行时出 bug |

#### 案例 2：双线程池 Sub-Agent

| 问题 | 答案 |
|------|------|
| 解决什么？ | Lead agent 不能等 sub-agent 完成才继续，否则用户要干等 15 分钟。所以用后台线程 + 轮询 |
| 放弃了什么？ | 放弃了真正的实时回调，改用 5 秒轮询。更简单但延迟更高 |
| 什么条件会崩？ | 3 个 sub-agent 都跑满了，第 4 个会排队。如果前 3 个都卡死（网络问题），整个系统卡住 |

#### 案例 3：Harness/App 分层

| 问题 | 答案 |
|------|------|
| 解决什么？ | deerflow-harness 包要能开源发布，app 层是业务定制。不分层的话，改业务代码可能破坏框架 |
| 放弃了什么？ | 开发体验 — 跨层调试时要跳两个包，import 路径更长 |
| 什么条件会崩？ | 有人在 harness 里 import app，CI 能拦住，但本地开发到提交前才发现 |

**关键发现：这三个问题主要靠读设计文档和边界测试，不是逐行读代码。**

## 二、判断 AI 代码对不对：四个检查点

你不需要懂每行代码，检查这四件事就够了：

### 1. 它理解上下文了吗？

AI 最常见的错误是**局部正确但全局错误**。

```python
# AI 可能写的
def process_message(msg):
    result = llm.invoke(msg)
    return result

# 问题：没有处理流式、没有 token 计数、没有中间件、没有错误重试
# 在 DeerFlow 的上下文里这完全是错的
```

**检查方法**：看它有没有读同目录下的其他文件。如果它没看过 `thread_state.py` 就写中间件，大概率会漏字段。

### 2. 它的假设对吗？

AI 默认假设"一切正常"。但它不知道你的项目：

- config.yaml 的字段可能改名了
- 某个中间件的执行顺序不是它以为的
- DeepSeek 的 reasoning_effort 不能和 thinking: disabled 并存

**检查方法**：找出代码里的隐式假设，逐个验证。比如 AI 写 `asyncio.get_event_loop()`，但项目用 `uvloop`，这就是隐式假设错误。

### 3. 错误处理呢？

AI 写的代码 80% 的情况是"happy path 正确，sad path 炸了"。

**检查方法**：只看 except/catch 分支和 if-not-exists 分支。问自己："如果这里返回 None / 抛异常 / 超时，会怎样？"

### 4. 它引入新依赖了吗？

AI 喜欢用自己训练数据里熟悉的库，而不是项目已有的。

**检查方法**：看 import。如果出现项目没用过的新库，先查 `pyproject.toml` 确认是不是已有依赖。

## 三、AI 写什么 vs 你学什么

### 让 AI 写的（胶水代码）

- Shell 脚本、Dockerfile、docker-compose
- CI/CD 配置
- Makefile（大部分）
- CRUD 接口
- 格式化、重构、重命名

### 你必须懂的（架构决策）

- 中间件链的设计模式
- Harness/App 分层的架构决策
- 流式输出为什么是两条路径
- Sub-Agent 的线程池设计
- Memory 的去抖动 + 原子写入
- Sandbox 的虚拟路径隔离

### 判断标准

问自己一个问题：**"如果需求变了，这个代码要改几个地方？"**

- 只改 1 个地方 → 胶水代码，AI 写
- 要改 3 个以上地方，而且要理解为什么 → 架构决策，你必须懂

## 四、知识盲区怎么补

### 原则 1：带着问题学，不是带着书本学

```
❌ 错误路径：先学完 asyncio 文档 → 再看 DeerFlow 的异步代码
✅ 正确路径：读 DeerFlow 的 subagent executor → 遇到 asyncio.gather 不懂 → 查 10 分钟 → 继续读
```

### 原则 2：只补"理解层"，不补"实现层"

你需要理解 asyncio.gather 是并行执行多个协程，不需要能手写一个事件循环。

需要理解的程度：

| 概念 | 你需要理解的 | 不需要懂的 |
|------|-------------|-----------|
| `docker compose up --build -d` | build 构建镜像、-d 后台运行 | overlay filesystem 实现 |
| `asyncio.gather` | 并行等待多个协程 | epoll/kqueue 的区别 |
| `awk` 状态机 | 它在解析 YAML 缩进 | 怎么从零写一个 |
| `uv sync` | 安装依赖到虚拟环境 | uv 的 Rust 实现细节 |

### 原则 3：建一个"够用知识库"

不是系统学习，是**按需积累**。每次遇到不懂的，记一句话：

```
asyncio.gather → 并行执行多个协程，有一个挂了全挂（除非 return_exceptions=True）
docker bind-mount → 主机文件不存在会变成目录，所以脚本要先 touch
awk → bash 里处理文本的工具，比 sed 强但比 Python 弱
set -e → bash 脚本里任何命令失败就退出，防止错误传播
```

## 五、AI 识别代码能力

看到一段代码，快速判断是不是 AI 生成的：

| 特征 | 说明 |
|------|------|
| 过度防御性编程 | 每个分支都有友好提示，注释密集但价值低 |
| 结构过于工整 | 每个 case 分支格式完全一致，人写通常更随意 |
| 注释说明显而易见的事 | 如 `# Main command dispatcher` 在 `case` 上面 |
| 用复杂方式解决简单问题 | 如用 awk 状态机解析 YAML，因为 AI 不知道 bash 里没有好用的 YAML 库 |
| 缺少项目特定知识 | 不知道项目约定、不知道已有工具函数 |

**识别出来不是为了鄙视，而是为了更仔细地审查。** AI 代码的 happy path 通常正确，sad path 需要你把关。

## 六、安全背景的差异化优势

架构决策里最重要的是**"它在什么条件下会崩"** — 这恰好是安全工程师最擅长的。

### 普通开发者 vs 安全工程师看代码的区别

```python
# sandbox_middleware.py 的 acquire/release 模式

# 普通开发者想：它怎么工作的？acquire 分配资源，release 释放。
# 安全工程师想：如果 acquire 成功但 release 没被调用（进程被 kill），sandbox 会不会泄漏？
```

```python
# memory 的原子写入（temp file + rename）

# 普通开发者想：先写临时文件再改名，防止写入一半断电。
# 安全工程师想：temp file 的权限是什么？会不会被其他用户读到？rename 是原子操作吗（同文件系统才是）？
```

```python
# 工具动态加载 resolve_variable("deerflow.sandbox.tools:bash_tool")

# 普通开发者想：通过字符串加载代码，很灵活。
# 安全工程师想：如果 config.yaml 被篡改，能不能加载任意代码？有没有白名单校验？
```

**这种"攻击者思维"是稀缺的。** AI 写代码不会想这些。你能想这些，你就比 AI + 普通开发者都值钱。
